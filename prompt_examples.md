# AI 提示词实战手册

> 每个模块背后的 AI 对话过程：你该怎么问，AI 怎么回答，怎么追问迭代

---

## 通用原则：好的提示词长什么样

在开始之前，先记住一个公式：

**好的提示词 = 角色 + 场景 + 具体要求 + 约束条件**

| 要素 | 差的例子 | 好的例子 |
|------|----------|----------|
| 角色 | （没说） | "你是一个 Python 教学专家" |
| 场景 | "解释 pandas" | "我是奶茶店店长，用我的销售记录来解释 pandas 筛选" |
| 具体要求 | "写段代码" | "写一段 30 行以内的代码，覆盖 5 种筛选方式" |
| 约束条件 | （没说） | "注释用中文，每个操作用生活类比解释" |

## 模块 1：pandas 入门 — 用生活类比学数据筛选

### 第一轮提示：让 AI 设计类比

```
你是一位擅长用生活例子教学的 Python 老师。

我在做一个培训材料，想用「奶茶店记账」的场景来教同事们学 pandas 数据筛选。
请帮我：

1. 先创建一周的奶茶销售数据（包含日期、饮品名、杯数、单价、是否加糖）
2. 用 5 个递进的例子展示不同的筛选方式：
   - 单条件筛选（比如只找珍珠奶茶）
   - 数值比较（比如找爆款——杯数 > 30）
   - 组合筛选 AND（比如加糖的珍珠奶茶）
   - 组合筛选 OR
   - 用 .query() 方法
3. 每个操作都用一句话的生活类比来解释原理

代码要求：纯 Python 文件，注释用中文，控制在 80 行以内。
```

### AI 可能的回应思路

AI 通常会生成一个完整的 DataFrame + 多个筛选示例。你需要检查：
- 数据是否直观（奶茶店的例子大家都能理解）
- 类比是否贴切（`df[条件]` → "用荧光笔在纸质表格上标注"）
- 代码是否能直接运行

### 第二轮追问：优化类比

```
类比写得不错，但有些解释还是偏技术。请帮我检查：
- 每个筛选操作的注释里，是否都有一个「像……一样」的比喻？
- 把布尔索引的 True/False 解释为「打勾/不打勾」
- 在最后加一个总结框，把 pandas 操作和日常动作对应起来
```

### 第三轮追问：让 AI 变成学生来检验

```
现在反过来：我来写一段 pandas 筛选代码，你扮演一个完全不懂编程的奶茶店老板，
用你的理解来复述我的代码在做什么。如果我写得不对，用老板能理解的方式告诉我哪里有问题。
```

## 模块 2：debug 练习 — AI 生成含 Bug 代码

### 第一轮提示：构造练习

```
你是一位 Python 教学专家。请帮我设计一段「找 Bug」练习代码：

场景：学生成绩统计（计算平均分、最高分，找不及格的学生）
要求：
- 代码看起来像初学者写的，逻辑上似乎没问题
- 故意埋入 5 个不同类型的 Bug：
  1. 数据类型错误（比如数字存成了字符串）
  2. 空值处理（某个字段是 None）
  3. 数据校验缺失（分数超出合理范围）
  4. 缺失字段访问（KeyError）
  5. 边界条件（空列表导致除零）
- Bug 之间要有依赖关系——修复了一个才能发现下一个
- 不要在代码里标注 Bug 的位置

给我纯净版代码（不含答案），另外再给我一份教师版（带注释和修复方案）。
```

### 第二轮追问：调整难度

```
这段代码对初学者来说可能太难了。请帮我：
1. 把 5 个 Bug 减少到 3 个，只保留最常见的类型（类型错误、空值、除零）
2. 在代码开头加一个运行说明
3. 在代码末尾加「提示」区块——不直接给答案，而是引导思考方向
```

### 第三轮追问：让 AI 做代码评审

```
我修复了所有 Bug，这是我的代码：
（粘贴修复后的代码）

请帮我做 Code Review：
1. 我的修复方式是否合理？有没有更 Pythonic 的写法？
2. 有没有我遗漏的潜在问题？
3. 如果要加单元测试，你会建议测试哪些 case？
```

## 模块 3：Flask 图书管理 — AI 生成项目框架

### 第一轮提示：描述需求

```
我需要一个简单的图书管理 API，用 Python Flask 实现：

功能需求：
- 增删改查图书（书名、作者、出版年份、ISBN）
- 用 SQLite 存储
- 支持按作者名筛选
- 合理的错误处理（缺少字段、ISBN 重复、ID 不存在）

技术约束：
- 单文件实现（不需要分多个 .py 文件）
- 不需要用户认证
- 代码控制在 150 行以内
- 注释用中文

请生成完整可运行的代码，包含数据库初始化逻辑。
```

### 第二轮追问：学习架构设计

```
代码我已经跑通了。现在我想学习你设计这个应用的思路：
1. 为什么用 SQLite 而不是 JSON 文件？
2. 路由是怎么设计的？RESTful 是什么意思？
3. 错误码 400/404/409 分别在什么情况下返回？
4. 如果我想加一个「按 ISBN 精确查找」的功能，应该改哪里？
```

### 第三轮追问：迭代功能

```
请在现有代码基础上，帮我加上这些功能：
1. 分页查询：GET /books?page=1&per_page=10
2. 搜索功能：GET /books?q=关键词（同时搜索书名和作者）
3. 在首页加一个 API 使用说明的 JSON 返回

每个新功能请解释你在哪里加了什么代码、为什么这样设计。
```

## 模块 4：R dplyr 管道操作

### 第一轮提示

```
你是一位 R 语言数据科学教练。

我有一份电商订单数据（包含订单ID、客户、产品、品类、金额、数量、日期），
我想用 dplyr 的管道操作 %>% 来完成以下分析：

1. 按品类 × 月份 汇总销售额
2. 排除小额订单（金额 < 10）
3. 按销售额排名
4. 添加客户城市信息（通过另一张客户表 join）
5. 计算每个客户在其所属级别（VIP/普通）中的排名

请用一条 %>% 管道完成，每一步都加中文注释解释这步在做什么。
同时在代码后面给出「管道拆解表」——每步操作对应一个日常动作的类比。
```

### 追问技巧

```
这条管道太长了，我想拆成 3 个短管道来讲授。
请帮我在合适的地方断开，给每个短管道取一个业务含义的名字（比如「数据清洗」「汇总统计」「客户分层」）。
```

## 模块 5：R flextable 条件格式

### 第一轮提示

```
我有一份销售团队的业绩数据（姓名、销售额、目标额、完成率），
我想用 R 的 flextable 包生成一份 Word 文档中的专业表格。

要求：
1. 基础版：带表头美化、斑马纹、数字格式化
2. 进阶版：完成率 >= 100% 标绿，< 80% 标红
3. 高级版：销售额列用颜色渐变（越高越深）

请生成 R 脚本，三个版本分别导出为 Word 文件。
代码中请注释每个 flextable 函数的作用。
```

### 追问

```
现在帮我把这三个版本封装成一个函数 make_sales_table(data, style = "basic")，
以后只需要一行代码就能生成不同风格的表格。
```

## 模块 6：R Markdown / Quarto 报告模板

### 第一轮提示

```
我需要一份 R Markdown 报告模板，用于电商季度销售分析：

报告结构：
1. 执行摘要（关键数字）
2. 销售趋势（折线图 + 月度对比表）
3. 区域分析（柱状图 + 健康度诊断表）
4. 品类分析（饼图 + 交叉热力图）
5. 结论与建议

技术要求：
- 支持中文
- 输出 PDF 格式
- 用参数化报告设计（params），方便不同季度复用
- 所有图表和表格用模拟数据生成，确保能直接渲染

请给我完整的 .Rmd 文件。
```

### 追问：迁移到 Quarto

```
这份 R Markdown 模板很好。现在请帮我把它迁移到 Quarto (.qmd)：
1. YAML 头部改为 Quarto 格式
2. 用 Quarto 的新特性（比如 code-fold、callout）
3. 同时支持 HTML 和 PDF 输出
请说明 R Markdown 和 Quarto 的主要区别。
```

## 模块 7：Rust 所有权 — 生活类比

### 第一轮提示

```
你是一位 Rust 教学专家。

我想用生活中的例子来理解 Rust 的「所有权」概念。
请帮我设计 5-6 个类比，分别对应：
- 所有权 (Ownership)
- 移动 (Move)
- 借用 (& 引用)
- 可变借用 (&mut)
- 生命周期 (Lifetime)
- Copy vs Clone

要求：
1. 每个类比都是日常生活中能遇到的场景
2. 用 Rust 代码演示每个类比，注释里写清楚对应关系
3. 最后用一段代码展示「三大铁律」

代码要能直接编译运行。
```

### 追问

```
这些类比很好。但「生命周期」的类比还不够直观。
请再用 2-3 个不同的生活例子来解释生命周期，
特别是「引用不能比它指向的数据活得更久」这一点。
```

## 模块 9：Rust Tauri 桌面应用

### 第一轮提示

```
我想用 Rust + Tauri 做一个简单的备忘录桌面应用。

后端需求（Rust）：
- 备忘录数据结构：ID、标题、内容、标签列表、创建时间、是否置顶
- CRUD 操作 + 搜索 + 标签筛选
- 用 Mutex 保证线程安全
- 数据存内存即可（不需要持久化）

前端需求（HTML/CSS/JS）：
- 卡片式布局
- 搜索栏 + 标签筛选
- 新建/编辑/删除表单
- 置顶功能

请生成完整的项目结构，包含 Cargo.toml、main.rs 和 index.html。
```

### 追问

```
项目跑通了！现在我想理解前后端是怎么通信的：
1. invoke() 这个函数在底层做了什么？
2. Rust 端的 #[tauri::command] 宏是怎么工作的？
3. 如果我想把数据持久化到本地文件，应该怎么改？
```

## 日常工作中的 AI 提示词模板

除了培训材料，以下是日常业务中常用的提示词模式：

### 数据分析

```
我有一份 [描述数据] 的 CSV 文件，列包括 [列名列表]。
请帮我用 [Python pandas / R dplyr] 完成以下分析：
1. [具体分析目标]
2. 生成可视化图表
3. 用一段话总结你发现的关键洞察
```

### 代码审查

```
请帮我审查这段代码，重点关注：
1. 潜在的 Bug 或边界情况
2. 性能问题
3. 可读性改进建议
不需要修改代码，只给出建议。
```

### 文档撰写

```
请帮我写一份 [报告/邮件/方案]：
- 读者是 [谁]
- 目的是 [什么]
- 关键信息包括 [要点列表]
- 语气 [正式/轻松/技术]
- 控制在 [字数] 以内
```

### 学习新概念

```
我想理解 [概念名称]。
请用以下结构帮我：
1. 一句话解释（给非技术人员听的）
2. 一个生活中的类比
3. 一个最小代码示例
4. 一个常见误区
```
