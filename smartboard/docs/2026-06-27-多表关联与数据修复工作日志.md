## 一、后端数据分析架构说明

### 1.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面框架 | Tauri 2 | Rust 原生后端 + Web 前端 |

| CSV 解析 | PapaParse（前端）/ Polars CSV reader（后端） | |

### 1.2 后端命令体系

```
src-tauri/src/commands/
├── loader.rs           # 文件加载（CSV/Excel → Polars DataFrame）
├── excel_selection.rs  # Excel 多 sheet 选择与加载
├── merge.rs            # 数据集 JOIN / CONCAT
├── groupby.rs          # 分组聚合（sum/mean/count/min/max）
└── mod.rs              # 模块导出
```

**数据流：**
```
Excel/CSV 文件
  → calamine/Polars 读取
  → Polars DataFrame（全局状态 GLOBAL_DF）
  → df_to_payload() 序列化为 ChartPayload JSON
  → Tauri invoke 返回前端
  → Pinia Store（data-store）管理多表状态
  → 前端聚合/筛选/公式引擎消费
```

### 1.3 双模式运行

- **Tauri 桌面模式**：Rust/Polars 后端处理（高性能，支持大文件）
- **浏览器降级模式**：前端 SheetJS + PapaParse + JS 聚合（开发调试用）

## 二、P0 — 数据正确性修复（Critical）

### 2.1 Excel 日期序列号未转换

**问题：** Excel 日期以序列号存储（如 `45309` ≈ 2024-01-18），calamine 读取后直接 `.to_string()` 输出原始数字，随后被 `auto_cast_numeric` 转为 Float64，前端永远收不到日期字符串。

**修复文件：**

| 文件 | 修改 |
|------|------|
| `src-tauri/src/commands/loader.rs` | 新增 `excel_serial_to_date()`、`is_date_column_name()`、`cell_to_string()` 三个函数；`load_excel_first_sheet` 预计算日期列掩码并应用转换 |
| `src-tauri/src/commands/excel_selection.rs` | `load_sheet_impl` 复用上述公共函数，修复按 sheet 单独加载时的日期问题 |

**日期转换逻辑：**
- 列名启发式：`date/日期/时间/time/年/月/日/出生/birth/创建/到期` → 标记为日期列
- `Float` 值在 30000–80000 区间的整数 → 调用 `excel_serial_to_date()`
- `DateTime(ExcelDateTime)` → 优先使用 calamine 内置 `as_datetime()`（自动处理 1900/1904 日期系统差异）
- 手动处理 1900 闰年 bug：serial ≥ 61 需减 1 天偏移
- 输出统一为 `YYYY-MM-DD` 格式

### 2.2 仪表盘数据表列显示与配置页不同步

**问题：** 关联多表后，Dashboard 页面的数据表列数/列名与 Config 配置页不一致。

**根因：** 三处使用了不同的列头来源：
- ConfigView `allHeaders` 使用 `allFieldOptions`（始终带 `"表名."` 前缀）
- DashboardView `allColumns` 使用 `dataSet.headers`（仅活跃表）
- `buildEffectiveDS` 合并时仅冲突列加前缀

**修复：**
- `data-store.ts`：新增 `effectiveHeaders` 计算属性，与 `buildEffectiveDS` 一致的合并逻辑
- `preview-store.ts`：暴露 `effectiveHeaders`
- `ConfigView.vue`：`allHeaders` 改用 `previewStore.effectiveHeaders`
- `DashboardView.vue`：`allColumns`/`allHeaders` 改用 `previewStore.effectiveHeaders`
- `config-store.ts`：`generateAutoTable()` 使用 `effectiveHeaders`

## 三、P1 — 跨表关联功能完善（High）

### 3.1 配置页各区块不支持关联表列选择

**问题：** 关联表格后，配置页的筛选、时间切片、全局格式、KPI、图表等区块的列选择器只能看到活跃表的列，无法选择关联表的列。

**修复文件：**

| 文件 | 修改 |
|------|------|
| `src/stores/data-store.ts` | 新增 `getEffectiveClassification(col)` — 按优先级遍历所有表查找列分类（活跃表 → 前缀跨表引用 → 全表扫描） |
| `src/stores/config-store.ts` | `_effRole()`、`_autoBase()`、`generateAutoKpis()`、`generateAutoCharts()`、`generateAutoTable()` 全部改用跨表列头 + 跨表分类查询 |
| `src/views/ConfigView.vue` | `effRole()`、`numericCols`、`dimensionCols`、`allMetricCols`、`dateCols`、`isNumericCol()` 全部支持跨表列 |
| `src/views/DashboardView.vue` | `isNumericCol()`、`allMetricCols` 支持跨表列 |

**分类查询优先级：**
```
getEffectiveClassification(col):
  1. 活跃表 dataSet.classifications[col]
  2. 带前缀解析（如 "表B.金额" → 解析前缀后查对应表）
  3. 遍历所有 tables 查找同名列
  4. null（未找到）
```

## 六、数据架构关键设计

### 6.1 跨表列命名一致性

```
effectiveHeaders（data-store） ≡ buildEffectiveDS.mergedHeaders（preview-store）

规则：右表列仅在名称与左表冲突时添加 "表名." 前缀
  例：左表[项目, 金额] + 右表[项目, 负责人]
  → effectiveHeaders = [项目, 金额, 负责人]     （"项目"跳过，"负责人"无冲突）
  → effectiveHeaders = [项目, 金额, 表B.项目]   （"项目"冲突时加前缀）
```

### 6.2 分类查询统一入口

所有列分类查询应通过 `dataStore.getEffectiveClassification(col)` 而非直接访问 `dataSet.classifications[col]`，确保跨表列的分类信息可获取。

### 6.3 数据行合并

`previewStore.applyFilters()` 和 `buildSpec()` 内部调用 `buildEffectiveDS(ds)` 将关联表数据左连接合并到行数据中，后续 `r[col]` 即可直接访问跨表列值。
