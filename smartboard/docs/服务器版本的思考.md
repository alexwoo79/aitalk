这是一个很好的架构演进思路。基于刚才对整个项目的深度分析，我来回答这两个问题。

---

## 能否变为 HTTP 网络应用？

**可以，而且改造路径比较清晰。** 项目的架构分层其实已经为此做了不少铺垫。

当前架构的核心耦合点有三个层次，改造难度递增：

**第一层：IPC 桥接层（最容易改）。** `use-rust-bridge.ts` 把 29 个 Tauri 命令封装在一个文件里，所有前端调用都走 `invoke('command_name', params)` 这个统一模式。改为 HTTP 只需把这 29 个 `invoke` 替换为 `fetch('/api/command_name', { body: params })`。这个文件的抽象设计让改动集中在一个地方。

**第二层：文件系统交互（中等工作量）。** 11 个前端文件直接使用了 `@tauri-apps/plugin-dialog`（文件选择/保存对话框）和 `@tauri-apps/plugin-fs`（文件读写）。在 Web 环境下需要替换为浏览器原生方案：文件上传用 `<input type="file">`，文件下载用 `Blob` + `URL.createObjectURL`。好消息是 `save/entry.ts` 已经证明了这个路径可行——它完全用浏览器原生 API 实现了 CSV 导出和主题存储，零 Tauri 依赖。

**第三层：Rust 后端改造（核心工作量）。** 4,136 行 Rust 命令代码（Polars 数据处理、HTTP 请求、SQL 驱动、飞书 API）背后的业务逻辑是框架无关的。改造方案有两条路：

- **方案 A：Rust 后端 + Axum/Actix-web HTTP 框架。** 把 `#[tauri::command]` 替换为 Axum 的 `handler`，`invoke` 参数映射为 JSON body 解析。Polars、reqwest、rusqlite 这些依赖在 HTTP 服务中完全可用，几乎不用改业务逻辑。
- **方案 B：编译为 WASM。** Polars 有 WASM 支持，但 MySQL/Postgres 驱动和文件系统操作需要额外适配，限制较多。

方案 A 更实际。

---

## 能否变为局域网多用户工具？

**可以，但需要解决状态隔离问题——这是最大的架构障碍。**

当前的 `state.rs` 用全局静态变量管理所有状态：

```rust
pub static GLOBAL_DF: Lazy<RwLock<Option<Arc<DataFrame>>>> = ...;
pub static DATASET_REGISTRY: Lazy<RwLock<Vec<RuntimeDataset>>> = ...;
```

这是典型的单用户桌面设计——一个进程、一个窗口、一份数据。如果多个用户同时访问 HTTP 服务，所有人会读写同一份 DataFrame，数据互相覆盖。

改造方向是**把全局状态改为会话级状态**：

```rust
// 每个用户会话持有独立的 DataFrame 集合
struct SessionState {
    dataframes: HashMap<String, Arc<DataFrame>>,
    active_id: Option<String>,
    // ...
}

// 用 session_id 隔离
type SessionStore = Arc<DashMap<String, SessionState>>;
```

每个命令从接收 `session_id` 开始，通过 session store 获取对应用户的数据。这需要对所有 Rust 命令的签名做调整，但逻辑本身不用改。

## 实际可行性评估

从投入产出比来看，分三个阶段比较合理：

**阶段一：单用户 Web 版（1-2 周）。** 启动 Axum HTTP 服务，替换 Tauri 命令为 HTTP handler，前端 `use-rust-bridge.ts` 改为 fetch 调用，文件交互改为浏览器原生方案。状态保持全局单份（一个浏览器标签对应一个服务实例）。这已经可以作为一个"打开浏览器就能用"的 Web 应用了。

**阶段二：局域网单实例版（额外 1 周）。** 加上简单的 session 隔离（内存中的 HashMap + session cookie），支持局域网内多人同时使用各自的数据。Rust 的 Polars 处理性能远超 JS，一台普通服务器同时支撑 10-20 个用户的数据分析完全没有问题。

**阶段三：生产级多用户版（额外 2-4 周）。** 加上用户认证、数据持久化（DataFrame 序列化到磁盘/数据库）、并发限制、资源配额等。

---

## Rust 计算能力是否足够？

完全足够。Polars 本身就是为高性能设计的——它底层用了 Arrow 列式内存格式，支持多线程并行计算。从之前的性能分析看，当前 Rust 后端只做了基础聚合和筛选，大量算力其实闲置。一个 4 核服务器同时处理 10-20 个用户的 10 万行级数据集，对 Polars 来说很轻松。真正可能成为瓶颈的是内存——每个用户的 DataFrame 都要驻留在内存中，20 个用户各 100MB 数据就需要 2GB+ 的内存预算。

总结来说，这个项目的架构已经为 Web 化做了不少准备（`isTauri()` 降级、`save/entry.ts` 纯 Web 导出、核心计算模块框架无关），从桌面应用演进到局域网工具是一条可行的路径。
