#### 1.1 相关性热力图（Correlation Heatmap）

**分析价值**：一键看清所有数值指标的相互关系。

```
            revenue  cost  orders  ad_spend  returns
revenue      1.00   0.95   0.98    0.72     -0.15
cost         0.95   1.00   0.93    0.68     -0.10
orders       0.98   0.93   1.00    0.70     -0.12
ad_spend     0.72   0.68   0.70    1.00      0.05
returns     -0.15  -0.10  -0.12    0.05      1.00
```

**技术方案**：
- 算法：皮尔逊相关系数矩阵
- 图表：ECharts Heatmap + visualMap
- 色阶：红（正相关）→ 白（无关）→ 蓝（负相关）

#### 1.2 帕累托图（Pareto Chart）

**分析价值**：比十分位更直观地展示"二八效应"。

```
柱状（降序） + 累计折线（第二Y轴）
→ 一眼看出前几项贡献了 80%
```

**技术方案**：
- 双 Y 轴：左轴柱状值，右轴累计百分比 0~100%
- 80% 参考线
- 复用现有 aggregate 逻辑

#### 1.3 异常值检测（Outlier Detection）

**分析价值**：自动发现数据中的离群点。

```
IQR 方法：
  Q1 = 25百分位, Q3 = 75百分位
  IQR = Q3 - Q1
  异常下界 = Q1 - 1.5×IQR
  异常上界 = Q3 + 1.5×IQR

输出：在现有图表上用散点/颜色高亮异常数据点
```

**技术方案**：
- 新增 `computeOutliers()` 函数到 `analysis.ts`
- 在各图表 option 中增加 markPoint 标注
- 数据表异常行高亮

### A1. 高级分析移出主线程（#6）

**现状**：`analysis.ts` 中 `computeTimeseries` / `computeDeciles` / `computeClusters` 纯同步 JS，大数据集下阻塞 UI。

**方案对比**：

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|:--:|
| A. Web Worker | 不阻塞 UI，实现简单 | 数据传输需序列化，调试困难 | 快速实现 |
| B. 移到 Rust backend | 最高性能，复用 Polars | 算法需重写，工作量大 | 长期 |
| C. `requestIdleCallback` 分片 | 不改架构 | 复杂度高，仍卡顿 | ❌ |

**建议**：Phase 1 用 Web Worker（`new Worker('analysis.worker.ts')`），Phase 2 移植 Rust。

**预估**：
- Web Worker：2 天（新建 `analysis.worker.ts`，`comlink` 通信）
- Rust 移植：5 天（Polars 时序/聚类算法实现）

### A2. CPU 工作移出 async 线程（#16）

**现状**：所有 `#[tauri::command]` 的 Polars 操作直接运行在 tokio async 线程池，阻塞其他请求。

**方案**：

```rust
// 之前：直接在 async 线程执行
pub async fn compute_dashboard(...) -> ApiResult<...> {
    let df = get_active_df()?;          // 阻塞 tokio 线程
    let filtered = apply_filters(&df)?;  // 阻塞 tokio 线程
}

// 之后：spawn_blocking 移到专用线程池
pub async fn compute_dashboard(...) -> ApiResult<...> {
    tokio::task::spawn_blocking(move || {
        let df = get_active_df()?;
        let filtered = apply_filters(&df)?;
        // ...
    }).await?
}
```

**变更文件**：`compute.rs`、`groupby.rs`、`loader.rs`、`merge.rs`、`database.rs`、`datasource.rs`、`feishu.rs` 所有 `#[tauri::command]` 函数。

**预估**：1 天（所有 command 包裹 `spawn_blocking`）

### A3. 剪贴板/SQLite 去临时文件（#17）

**现状**：`paste_from_clipboard` 和 `load_sqlite_query` 将数据写入临时 CSV 文件再 Polars 读取：

```rust
fn paste_from_clipboard(text: String, has_header: bool) -> ... {
    let path = temp_csv_path("paste");
    fs::write(&path, &text)?;
    CsvReadOptions::default().try_into_reader_with_file_path(Some(&path))?;
}
```

**方案**：使用 `CsvReader::new(Cursor::new(text))` 从内存读取：

```rust
use std::io::Cursor;
let df = CsvReader::new(Cursor::new(text))
    .has_header(has_header)
    .finish()?;
```

**变更文件**：`datasource.rs`（`paste_from_clipboard`、`load_sqlite_query`）

**预估**：0.5 天

### A4. Lazy Frame 真正融合（#18）

**现状**：`.lazy()` 后紧跟 `.collect()`，优化器无机会融合多步操作：

```rust
// 当前：每步独立 collect
let df = df.clone().lazy().filter(...).collect()?;
let df = df.lazy().filter(...).collect()?;
```

**方案**：构建完整 lazy pipeline 再一次性 collect：

```rust
// 优化后：构建完整表达式链，一次 collect
let mut lazy = df.clone().lazy();
if has_dim { lazy = lazy.filter(dim_expr); }
if has_date { lazy = lazy.filter(date_expr); }
if has_search { lazy = lazy.filter(search_expr); }
let result = lazy.collect()?;
```

**变更文件**：`compute.rs`（`apply_dim_date_filters`）

**预估**：0.5 天（重构已部分完成，搜索和条件已在同一 pipeline）
