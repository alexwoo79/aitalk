# 计算指标体系设计方案

> 日期：2026-06-25  
> 状态：设计阶段（待实现）

## 一、背景与问题

### 当前架构的问题

1. **动态计算无缓存**：所有指标基于原始列实时聚合（sum/avg/count/min/max），每次图表渲染都重复遍历全量数据
2. **无派生指标概念**：无法定义"利润率 = 利润 / 收入 × 100"这样的计算指标，并在图表、KPI、数据表中统一引用
3. **KPI Formula 局限**：
   - 仅限 KPI 卡片使用，图表/数据表无法引用公式
   - 使用 `[0]` / `[1]` 位置占位符，不直观，变量顺序一变就断
   - 不可组合：无法基于"利润率"再派生出"毛利率占比"
4. **计算重复**：同一指标在不同图表中独立聚合，没有复用

### 目标

- 定义**计算指标（Computed Metric）**为"一等公民"
- 一处定义，KPI / 图表 / 数据表均可引用
- 支持指标间的组合依赖（DAG 拓扑）
- 统一格式化配置
- 兼容 Rust/Polars 后端和 JS 导出两种模式

## 二、类型定义

### ComputedMetric

```typescript
// src/types/metric.ts（新增）

/** 计算指标定义 */
export interface ComputedMetric {
  /** 唯一标识（同时也是"虚拟列名"） */
  name: string
  /** 显示标签 */
  label: string
  /** 计算表达式，支持引用变量的 alias 名称 */
  expression: string
  /** 参与的变量列表 */
  variables: MetricVariable[]
  /** 格式化配置（继承规则同原始列） */
  format?: MetricFormat
  /** 是否选中（KPI/图表中默认使用） */
  selected?: boolean
}

/** 计算指标变量 */
export interface MetricVariable {
  /** 数据来源：原始列名 或 其他 ComputedMetric.name */
  source: string
  /** 聚合方式（引用原始列时必填；引用计算指标时可选，默认继承） */
  agg?: 'sum' | 'avg' | 'count' | 'min' | 'max'
  /** 行级筛选条件 */
  filter?: string
  /** 表达式别名，供 expression 引用 */
  alias: string
}

/** 指标格式化 */
export interface MetricFormat {
  format?: 'number' | 'integer' | 'percent' | 'currency'
  unit?: 'yuan' | 'wan' | 'yi'
  prefix?: string
  decimals?: number
}
```

## 三、使用示例

### 定义计算指标

```typescript
const computedMetrics: ComputedMetric[] = [
  {
    name: 'profit_rate',
    label: '利润率',
    variables: [
      { source: '利润', agg: 'sum', alias: 'profit' },
      { source: '收入', agg: 'sum', alias: 'revenue' },
    ],
    expression: 'profit / revenue * 100',
    format: { format: 'percent', decimals: 1 },
  },
  {
    name: 'cost_rate',
    label: '成本率',
    variables: [
      { source: '成本', agg: 'sum', alias: 'cost' },
      { source: '收入', agg: 'sum', alias: 'revenue' },
    ],
    expression: 'cost / revenue * 100',
    format: { format: 'percent' },
  },
  {
    name: 'gross_margin',
    label: '毛利率',
    variables: [
      // 引用另一个计算指标！agg 省略表示继承原指标聚合
      { source: 'profit_rate', alias: 'pr' },
      { source: 'cost_rate', alias: 'cr' },
    ],
    expression: 'pr - cr',
    format: { format: 'percent' },
  },
  {
    name: 'yoy_growth',
    label: '同比增长率',
    variables: [
      { source: '收入', agg: 'sum', alias: 'current' },
      { source: '收入', agg: 'sum', filter: '日期 ~ 去年同期', alias: 'last_year' },
    ],
    expression: '(current - last_year) / abs(last_year) * 100',
    format: { format: 'percent', decimals: 1 },
  },
]
```

### 消费方引用

计算指标和原始列**完全平级**，通过 `name` 引用：

```typescript
// KPI — 引用计算指标
{ column: 'profit_rate', agg: 'sum', label: '综合利润率' }

// 图表 — 混合引用原始列 + 计算指标
{
  dimension: '地区',
  metrics: ['收入', 'profit_rate', 'yoy_growth'],
}

// 数据表 — 作为虚拟列展示
{ columns: ['地区', '收入', 'cost_rate', 'gross_margin'] }
```

## 四、计算引擎设计

### MetricEngine（JS 端，用于交互模式 + 导出 HTML）

```typescript
// src/core/metric-engine.ts（新增）

export class MetricEngine {
  private registry = new Map<string, ComputedMetric>()
  private cache = new Map<string, number>()

  register(m: ComputedMetric) { ... }
  registerAll(metrics: ComputedMetric[]) { ... }

  /** 判断一个名称是否为计算指标 */
  isComputed(name: string): boolean { ... }

  /** 拓扑排序依赖，返回计算顺序 */
  resolveOrder(names: string[]): ComputedMetric[] { ... }

  /**
   * 批量计算指标值
   * @param names 要计算的指标名列表
   * @param rows 原始数据行
   * @param filters 当前筛选条件
   * @param computeRaw 原始列聚合回调（解耦数据源）
   */
  compute(
    names: string[],
    rows: Record<string, string | number>[],
    filters?: FilterContext,
    computeRaw?: RawAggregator,
  ): Record<string, number> { ... }

  clearCache(): void { ... }
}

/** 原始列聚合回调接口 */
type RawAggregator = (
  column: string,
  agg: string,
  rows: Record<string, string | number>[],
  filter?: string,
) => number
```

### DAG 依赖解析流程

```
注册指标: profit_rate, cost_rate, gross_margin
请求计算: [gross_margin]

拓扑排序:
  1. profit_rate  (profit = sum(利润), revenue = sum(收入))
  2. cost_rate    (cost = sum(成本), revenue = sum(收入))  ← revenue 已缓存!
  3. gross_margin (pr = profit_rate, cr = cost_rate)       ← 引用计算指标

结果: { profit_rate: 25.3, cost_rate: 15.1, gross_margin: 10.2 }
```

### preview-store 改造

```typescript
// 改造前：仅支持原始列
function computeColumnValue(column, agg, rows, filter): number

// 改造后：自动判断原始列 / 计算指标
function computeMetricValue(
  name: string,
  rows: Record<string, string | number>[],
  context?: FilterContext,
): number {
  const engine = getMetricEngine()
  if (engine.isComputed(name)) {
    const result = engine.compute([name], rows, context)
    return result[name]
  }
  return computeColumnValue(name, 'sum', rows)
}
```

## 五、与 Rust/Polars 后端集成

### Rust 侧设计

```rust
// src-tauri/src/metric.rs（新增）

pub struct ComputedMetric {
    pub name: String,
    pub label: String,
    pub expression: String,
    pub variables: Vec<MetricVariable>,
    pub format: Option<MetricFormat>,
}
```

### 核心优势：Polars 表达式编译

Polars 的 `Expr` 系统天然支持表达式组合，可以做到**一次 IPC 往返完成所有计算**：

```rust
impl MetricRegistry {
    /// 递归构建 Polars 表达式
    fn build_expr(&self, name: &str) -> Expr {
        let m = self.find(name).unwrap();

        let var_exprs: Vec<(String, Expr)> = m.variables.iter().map(|v| {
            let expr = if self.has_metric(&v.source) {
                self.build_expr(&v.source)       // 递归：引用计算指标
            } else {
                col(&v.source).agg(v.agg())       // 原始列聚合
            };
            (v.alias.clone(), expr)
        }).collect();

        map_multiple(var_exprs, |args| eval_expression(&m.expression, args))
    }

    /// 批量计算（一次 Polars 查询）
    fn compute_all(&self, names: &[String], df: &DataFrame) -> Result<Vec<f64>> {
        let exprs: Vec<Expr> = names.iter()
            .map(|n| self.build_expr(n).alias(n))
            .collect();

        let result = df.clone().lazy().select(exprs).collect()?;

        Ok(result.row(0).unwrap().iter()
            .map(|s| s.try_get::<f64>().unwrap())
            .collect())
    }
}
```

优势：
- **零 JS 遍历**：所有计算在 Rust 侧向量化执行
- **一次 IPC**：前端只发送 `["profit_rate", "gross_margin"]`，返回 `[25.3, 10.2]`
- **Polars 自动并行**：多列聚合 + 表达式求值自动利用多核

## 六、导出 HTML（JS 回退模式）

`entry.ts`（save renderer）需要同步支持计算指标：

```typescript
// src/save/entry.ts

// 1. 从 __DATA__ 中读取 computedMetrics 注册表
const metricEngine = new MetricEngine()
if (DATA.computedMetrics) {
  metricEngine.registerAll(DATA.computedMetrics)
}

// 2. KPI 渲染时判断是否计算指标
function renderKpiCards(rows) {
  DATA.kpiSpecs.forEach((kpi, i) => {
    let value: number
    if (metricEngine.isComputed(kpi.column)) {
      value = metricEngine.compute([kpi.column], rows)[kpi.column]
    } else {
      value = computeAggValue(kpi.column, kpi.agg, rows)
    }
    // ... 格式化渲染
  })
}
```

## 七、对 chart-options.ts 的影响

**核心结论：`chart-options.ts` 无需改动。**

原因：图表构建函数（`buildBarOption` / `buildDoughnutOption` 等）消费的是**已经聚合好的数值**，不关心数据来源是原始列还是计算指标。

```
改造前的调用链路：
  chart-options.ts ← { labels: ['A','B'], values: [100, 200] }
                     ↑ 原始列聚合结果

改造后的调用链路（保持不变）：
  chart-options.ts ← { labels: ['A','B'], values: [100, 200] }
                     ↑ 原始列聚合结果 / 计算指标结果（调用方透明）
```

## 九、关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 表达式语法 | 简单算术：`+ - * / abs()` | 可被 `new Function()` 和 Polars `map` 双向编译 |
| 变量引用方式 | 命名 alias，非位置 `[0]` | 更可读，顺序无关，支持 DAG 组合 |
| 执行策略 | 批量计算 + 拓扑排序 | 避免重复计算，一次 IPC 完成 |
| 格式化 | 指标自带 format，支持 MetricDefaults 覆盖 | 复用现有 `fmtByChart` 逻辑 |
| 与原始列关系 | 平级：`metrics: ['收入', 'profit_rate']` | 最小化消费方改动 |
